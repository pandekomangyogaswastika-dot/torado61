## 2. RBAC matrix — how to extend (gate 3)

Edit `CASES` in `scripts/rbac_forensic_test.py`. Each case:
```python
("GET", "/api/<router>/<path>", body_or_None, {
    "SUPER_ADMIN": "ALLOW", "FINANCE_MANAGER": "DENY", ...
})
```
`ALLOW` = response is NOT 403 (200/404/409/422 all fine — we only assert the
permission gate did not block). `DENY` = must be exactly 403.

**Coverage goal:** at least one ALLOW and one DENY case per backend router that
exposes sensitive WRITE or financial/PII READ data.

## 3. IDOR / ownership — how to extend (gate 4)  ⭐ MOST IMPORTANT

**Why static is not enough:** `rbac_endpoint_guard_audit.py` flags ~59
"authenticated-only" endpoints (only `current_user`, no `require_perm`). Many are
SAFE because they enforce ownership INSIDE the handler — e.g. `/api/hr/leaves/{id}`
calls `_assert_can_view_employee_leave(...)`. A scanner CANNOT see that. **Only a
live cross-user probe proves it.**

Process for EACH authenticated-only endpoint that returns per-user/PII/financial data:
1. List candidates: `python scripts/rbac_endpoint_guard_audit.py` → the 🟠 section.
2. For each, decide: does it return data scoped to ONE user/entity?
3. Add a case to `CASES` in `scripts/idor_ownership_probe.py`:
   - `user_scoped` — the URL id IS a user/employee id.
   - `resource_scoped` — create a resource as owner, probe by its id, cleanup.
4. Re-run `python scripts/idor_ownership_probe.py` → expect `owner=allow`,
   `other=403`, `super=allow`, `missing=404`.

**Triage of the 59 authenticated-only endpoints (next agent — work this list):**
- `leave.py` summary/{employee_id}, /{leave_id} → ✅ FIXED + probed (ownership guard).
- `approvals.py` queue/pending/counts/quick-action/delegations → verify they only
  surface items the user is an eligible approver/owner of (not global).
- `admin.py` business-rules/* → entered via OUTLET_MANAGER's `admin.*` perms;
  confirm write ops require an admin perm (candidate P1).
- `daily_close.py` /{record_id}/reopen → confirm outlet-scope/permission.
- `master.py` /{entity}, /coa → likely read-reference data (low risk) — confirm.
- Remaining → classify read-reference (low) vs per-entity (must probe).

## 4. Frontend route-level RBAC guards (PHASE 2 — MANUAL/agent)

**Pattern (already proven in `portals/admin/AdminPortal.jsx`):**
```jsx
function permitted(user, needPrefix) {
  const perms = (user && user.permissions) || [];
  if (perms.includes("*")) return true;
  return perms.some((p) => p === needPrefix || p.startsWith(needPrefix + "."));
}
function Gate({ user, need, children }) {
  if (!permitted(user, need)) return <Navigate to="/no-access" replace />;
  return children;
}
// in portal: const g = (need, el) => <Gate user={user} need={need}>{el}</Gate>;
// <Route path="tax" element={g("finance.tax", <FinanceTaxHub/>)} />
```

**Steps per portal** (Finance, HR, Inventory, Outlet, Procurement, Owner, Executive/Reports):
1. Open `src/portals/<x>/<X>Portal.jsx` and map each `<Route>` to its perm namespace
   (use `backend/core/perms_catalog.py` + the section→reqPerm idea from
   `navigationSchema/admin.js`).
2. Wrap each sensitive route with `g("<namespace>", <Page/>)`.
3. Also add section-level `reqPerm` to `src/lib/navigationSchema/<x>.js` so the
   Sidebar hides what the route now blocks (consistency, no dead links).
4. **Verify via browser URL-tamper** (testing agent / screenshot tool): log in as a
   PARTIAL-access role (see leak map below) and hit a forbidden deep-link, e.g.
   FINANCE_STAFF → `/finance/tax`; expect redirect to `/no-access`.

**Leak-risk map (from `rbac_portal_access_audit.py`) — partial-access roles to test:**
- `/finance`: EXECUTIVE 7/39, FINANCE_STAFF 17/39, GM 4/39, OWNER 7/39

## 5. UI tours validation (PHASE 3 — MANUAL/agent browser)

Static gate (`audit_tours_v2.py`) only proves the `data-testid` LITERAL exists
*somewhere* in the codebase. It does NOT prove the element is RENDERED on the
tour's route at run time (route drift / conditional render = false positive).

**Procedure per tour** (≈9 tour files, ~98 logical tours, 429 step targets):
1. From `src/contexts/tour/tourMap.js`, get the route each tour starts on.
2. In the browser (testing agent / screenshot tool): log in (admin@torado.id),
   navigate to that route, start the tour, and confirm EACH step's target is
   visible + highlighted (no "target not found" / skipped step).
3. Fixed tours so far (do NOT regress): `executive-home`, `outlet-drilldown`,
   `brand-drilldown`.
4. Log any drift; fix by correcting the `target` selector or the route in
   `tourMap.js` / the tour file under `src/contexts/tour/tours/`.
